# Yang sudah sangat kuat

**1. README sudah judge-friendly.**
Ada “Verify in 60 Seconds”, command test, self-audit, dan CI gate. Ini bagus karena juri tidak perlu menebak cara menjalankannya.

**2. Fitur utama sudah lengkap.**
README mencantumkan SKILL.md validation, progressive disclosure, safety scanning, Solana fit, policy caps, strict CI mode, fail-under gate, report generation, dan self-audit.

**3. Policy caps sudah benar.**
Prompt injection, secret collection, opaque execution, priority manipulation, suspicious install, missing SKILL.md, dan no Solana content sekarang punya max score. Ini membuat scanner terasa lebih “security reviewer”, bukan sekadar penjumlah skor.

**4. Test suite naik besar.**
`test.mjs` sudah memakai `spawnSync`, menguji self-audit, policy caps, strict mode, fail-under, negation-aware scanning, benchmark fixtures, dan report generation.

**5. Self-audit 100/100 sudah jadi bukti kuat.**
Report self-audit menunjukkan 100/100, semua kategori pass, dan tidak ada must-fix / should-fix.

# Yang masih kurang untuk mengejar top 1/top 5

## 1. Benchmark masih synthetic, belum real submission

File benchmark kamu jujur menyatakan bahwa benchmark hanya memakai **5 local synthetic fixtures**, bukan real bounty submissions.  Ini aman secara kejujuran, tapi untuk top 1 masih kurang kuat.

Kompetitor top biasanya menunjukkan bukti “sudah dicoba pada kasus nyata”. Kamu perlu tambah:

```txt
examples/real-submission-smoke-test.md
examples/real-submission-smoke-test.json
```

Isinya tidak perlu menyerang peserta lain. Pakai 8–12 repo/PR publik secara anonim atau by link, lalu audit read-only. Tampilkan kategori umum: “strong progressive disclosure”, “missing install proof”, “safety clean”, “docs weak”, dan sebagainya.

Ini akan membuat sponsor berpikir: **“alat ini langsung bisa membantu kami review 247 submission.”**

## 2. Self-audit exclusion terlalu luas

Di `skillqa.mjs`, self-audit dideteksi hanya dari keberadaan `scripts/skillqa.mjs`.  Lalu ketika self-audit, scanner mengecualikan `scripts/`, `examples/`, `commands/`, `agents/`, `skill/`, `README.md`, dan beberapa file lain dari safety scanning.

Ini menyelesaikan false positive, tapi terlalu longgar. Secara security reviewer, juri bisa bertanya:

> “Kalau ada repo lain yang punya `scripts/skillqa.mjs`, apakah dia bisa memicu self-audit bypass?”

Solusinya: self-audit harus hanya aktif kalau path yang diaudit adalah root repo milik scanner sendiri, misalnya `resolve(skillPath) === resolve(__dirname, '..')`, bukan hanya “ada file `scripts/skillqa.mjs`”.

## 3. Belum ada `batch` command asli

README menjelaskan kebutuhan batch review, tapi CLI command utama masih `audit`, `score`, dan `report`. Untuk maintainer yang menghadapi 247 submissions, fitur killer-nya harus:

```bash
node scripts/skillqa.mjs batch ./submissions --out examples/batch-review.json --markdown examples/batch-review.md
```

Ini akan sangat menaikkan value. Sekarang scanner kamu menilai satu repo; Round 3 harus membuatnya bisa menilai banyak repo.

## 4. Solana fit masih keyword-based

README sudah jujur bahwa tool tidak melakukan semantic/LLM-based analysis dan tidak memverifikasi runtime behavior.  Itu bagus, tapi untuk top 1 kamu perlu tambahkan **evidence-based Solana fit**, bukan hanya keyword.

Contoh evidence check:

```txt
- Has Solana-specific examples?
- Has real command workflow?
- Mentions concrete Solana builder problem?
- Uses Solana stack terms in instructions, not only description?
- Has ecosystem-specific failure modes?
- Has explicit delegation boundaries with existing skills?
```

Jadi scoring tidak hanya “berapa banyak keyword Solana”.

## 5. README link Solana AI Kit perlu disamakan dengan bounty

README kamu sekarang mengarah ke `sendai-sf/solana-ai-kit`.  Sedangkan listing dan repo yang saya cek adalah `solanabr/solana-ai-kit`, dan repo itu menjelaskan Solana AI Kit sebagai production-ready Claude Code configuration dengan agents, commands, MCP, progressive loading, dan skill hub. ([GitHub][3])

Sebaiknya ganti link README ke:

```txt
https://github.com/solanabr/solana-ai-kit
```

Ini kecil, tapi untuk bounty sponsor, akurasi link penting.

# Prompt Round 3 untuk AI agent

Pakai prompt ini ke Opus/Sonnet:

You are a senior Solana AI Kit maintainer, security reviewer, and open-source engineer.

Repository:
[https://github.com/xDzaky/solana-skill-quality-gate](https://github.com/xDzaky/solana-skill-quality-gate)

Round 3 must push it toward top-5/top-1 by making it more useful for real maintainers reviewing 247+ bounty submissions.

Do not rewrite the project. Make targeted, high-leverage upgrades.

Required Round 3 upgrades:

1. Fix README Solana AI Kit link

* Replace any incorrect Solana AI Kit link with:
  [https://github.com/solanabr/solana-ai-kit](https://github.com/solanabr/solana-ai-kit)
* Keep references to the bounty and skill-bounty repo accurate.

2. Harden self-audit detection
   Current self-audit detection is too loose if it only checks whether `scripts/skillqa.mjs` exists.
   Change it so self-audit mode only activates when the audited path is exactly this repository root:

* `resolve(skillPath) === resolve(__dirname, '..')`
  or another equally strict identity check.
  Do not allow a random audited repo with `scripts/skillqa.mjs` to bypass safety scanning.
  Add tests:
* self-audit still passes for repo root
* a fixture containing `scripts/skillqa.mjs` but malicious SKILL.md is still flagged
* safety exclusions do not apply to arbitrary repos

3. Add batch review command
   Implement:

```bash
node scripts/skillqa.mjs batch <directory> --json --out examples/batch-review.json
node scripts/skillqa.mjs batch <directory> --markdown --out examples/batch-review.md
node scripts/skillqa.mjs batch <directory> --fail-under 80
```

Behavior:

* Treat each immediate child directory as one skill repo.
* Run the same audit logic on each.
* Output:

  * total skills scanned
  * pass/fail count
  * average raw score
  * average final score
  * policy-capped count
  * top risks
  * sorted results from lowest final score to highest
* Exit non-zero if any skill is below `--fail-under`.
* Do not execute audited scripts.
* Do not make network calls.

Add tests:

* batch command scans benchmark fixtures
* JSON output is valid
* markdown output is generated
* fail-under fails when one fixture is below threshold
* results are sorted by final score ascending

4. Add evidence-based Solana fit scoring
   Keep keyword scoring, but add an additional evidence layer so generic keyword stuffing is harder.

Evidence checks:

* SKILL.md has concrete Solana workflows, not only keywords
* focused skill files contain Solana-specific instructions
* README explains a real Solana builder problem
* examples or commands demonstrate Solana-specific use
* skill defines boundaries with existing ecosystem skills where relevant
* detects when keywords appear mostly in frontmatter/description but not in actual instructions

Add JSON output fields:

```json
"solanaEvidence": {
  "workflowEvidence": true,
  "focusedFilesEvidence": true,
  "readmeProblemEvidence": true,
  "examplesEvidence": false,
  "boundaryEvidence": true
}
```

Add report section:
`## Solana Fit Evidence`

Add tests:

* generic keyword-stuffing fixture remains capped or penalized
* excellent fixture shows multiple evidence signals
* self-audit shows strong evidence signals

5. Add SARIF output for GitHub Code Scanning compatibility

node scripts/skillqa.mjs report <path> --sarif --out skillqa.sarif

  This makes the tool look more production-grade and CI-friendly.

Add tests:

* SARIF has version "2.1.0"
* bad fixture creates SARIF results
* good fixture creates zero critical SARIF results
